## B カテゴリ: 「ローカル実行が必要」アンチパターンの撲滅詳細

waiting-user Issue の最大カテゴリは「ローカル実行が必要」（累計8件）だが、その多くは **L-094 型の desync**（インフラはクラウドで動くのに、ドキュメント/スキルが「ローカル必須」と古いまま）である。

### クラウドで利用可能なリソース表（プロジェクト例）

| リソース（プロジェクト例） | クラウドで使える理由 | 確認コマンド |
|---------|-------------------|------------|
| 外部投稿の認証情報（Cookie 等） | GitHub Variables（例: `*_COOKIES_JSON`）から取得 | `gh variable list -R kai-kou/claude-wiki-hub \| grep COOKIES` |
| オブジェクトストレージ（バックアップ） | アクセスキー等が GitHub Variables に登録済み | `gh variable list -R kai-kou/claude-wiki-hub \| grep -i storage` |
| レンダリング | クラウドで成功実績あり | レンダリングプロジェクトの存在確認 |
| 音声生成 | クラウドセッションで Docker 起動可能 | 音声パイプラインの実績 |
| 画像生成 | MCP 画像生成ツール（provider 切替）+ HTTP フォールバック | 画像生成スキルの SKILL.md |

### 「ローカル実行が必要」と Issue 化する前の必須手順

```
1. 上表で該当リソースがクラウドで使えないか確認
2. 実際にクラウドで実行を試行する（試さずに「ローカル必須」と書かない）
3. 失敗した場合のみ、エラー全文・試行コマンドを Issue に記録
4. problem-investigation-protocol.md の5ステップで根本原因を特定
5. それでもクラウド不可と判明したら、B カテゴリの実装 Issue を起票（ユーザー作業ではなくツール改修として扱う）
```

### 禁止パターン

```
❌ クラウド実行を試さずに「ローカル実行が必要」と user-work Issue を作成する
❌ 「Playwright だからクラウド不可」と決めつける（Cloudflare Browser Run 等の代替あり）
❌ 公開済みコンテンツの差し替えを「ローカルでお願いします」とユーザーに丸投げする
✅ まずクラウド実行 → 失敗証跡 → ツール改修 Issue（B カテゴリ）として自律的に解決経路に乗せる
```

## C カテゴリ: 自律処理ルール実装詳細

> **実行モデルの明示**: project-sync スキルは **Claude が GitHub MCP（`mcp__github__*`）・実在ツール・git で直接実行する単層構成** である（専用の Python オーケストレーターは持たない）。下表の「実装場所」は具体的な MCP 手順・実在ツールを指す。下表の事象は **プロジェクト例**。

| 事象（プロジェクト例） | 自律処理 | 実装場所 |
|------|---------|---------|
| バックログ候補ラベルに waiting-user 誤付与 | ラベルを自動除去 | project-sync SKILL.md Step 2（**Claude が MCP `issue_write` で実行**） |
| 週次/定例レポート系 Issue | Slack 通知済み + 7日経過 → 自動クローズ | project-sync SKILL.md Step 3（**Claude が MCP で実行**） |
| リサーチ・調査依頼が waiting-user 誤分類 | `status:waiting-claude` にリセット（担当スキルが自律実行） | project-sync SKILL.md Step 3（**Claude が MCP で実行**） |
| 環境変数未設定 | A 型チェックリスト → `gh variable set` で自律設定 | problem-investigation-protocol.md A 型チェックリスト（**ドキュメント手順**） |

新たに C カテゴリの事象を発見したら、本表に追記し、対応する自律処理を project-sync（SKILL.md の MCP 手順）あるいは該当スキルに実装する。**実装場所（どの MCP 手順・実在ツールか）を「実装場所」列に正確に記載すること**（L-094 desync 防止）。

## D カテゴリ: 外部要因の自律最小化ルール詳細

外部課金・BAN・プラットフォーム制約は設定変更（A-6）のみユーザー作業だが、**影響を自律的に最小化** する。下表は **プロジェクト例**。各プロジェクトが利用する外部 API・プラットフォームに読み替える。

| 外部要因（プロジェクト例） | 自律最小化策 | ステータス |
|---------|------------|-----------|
| 画像生成 API の請求ハード上限 | 枯渇前に Slack 予告。**実測検知時は `tools/api_depletion_alert.py` がハード上限到達を検知した瞬間に Slack 即通知（A-6）**。枯渇後は代替 provider の HTTP フォールバックで継続。代替エンジン降格は品質一貫性への影響評価が必要なため実装 Issue で慎重に検討 | 予告 + 実測即時通知 |
| 外部 API クレジット枯渇 | 予算トラッカーで枯渇前に予告（予測型）。**実測検知時は `tools/api_depletion_alert.py` が投稿/取得スクリプトのエラーパスでクレジット枯渇を検知した瞬間に Slack 即通知（A-6。推定残高が健全表示でも取りこぼさない）**。月初リセットを監視し復旧を自動確認 | 予告 + 実測即時通知 |
| GitHub Actions 課金超過 | **本プロジェクトは GitHub Actions を使わない**（`docs/rules/no-github-actions.md`）。CI は Claude Code ハーネス（self-reviewer のワークツリー隔離 e2e・フック・ルーティン）で代替し、Actions の課金そのものを発生させない | 恒久解消（Actions 不使用） |

「枯渇したので止めます」で終わらせず、**フォールバック経路で可能な限り処理を継続** することが D カテゴリの自律運用の要件。

## B/D カテゴリ恒久自律化バックログ

以下はツール/インフラ改修により「二度とユーザー確認不要」を実現する。実装は self-improvement-loop が消化モードで自律的に進める。下表は **プロジェクト例**。各プロジェクトの恒久自律化バックログを Issue 化して消化する。

| 項目（プロジェクト例） | 手段（CP-2 リサーチ済み） | 関連 Issue |
|------|------------------------|-----------|
| 公開済みコンテンツの差し替え（update）モード | 投稿スクリプトに既存コンテンツ置換モードを追加（認証情報はクラウド可） | （プロジェクトで起票） |
| 制作処理のクラウドレンダリング | クラウドセッション内でレンダリング → ストレージ転送を Claude が自律実行（**GitHub Actions は使わない**・`no-github-actions.md`） | （プロジェクトで起票） |
| 課金監視の自動予告・フォールバック | 各 API のコスト/クレジット監視をスケジュールタスクに統合・枯渇前予告 | （プロジェクトで起票） |
